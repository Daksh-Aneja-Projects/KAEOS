# KAEOS — L10 Independent Audit
**Date:** 2026-07-19 · **Method:** Static review only (Docker down) · **Reviewers:** 6 specialist auditors (backend architecture, security & multi-tenancy, frontend, AI/agent logic, data layer, claims-vs-reality), findings cross-checked against the source tree.

---

## 1. Verdict

KAEOS is **substantially more real than a typical "ambitious startup" codebase.** The load-bearing spine — auth, per-tenant RLS, LLM gateway with live cost metering, a genuine 6-gate agent pipeline with *blocking* HITL, and a frontend that is 38/39 pages actually wired to the backend — is real, security-conscious engineering. The E2E suite is live and anti-permissive. Prior-audit critical bugs (an IDOR across every domain router, a flagship `SyntaxError`) are genuinely fixed, and the engineering docs are unusually honest about their own holes.

**But it is not production-ready, and three classes of problem stand between it and that claim:**

1. **A small number of true security dealbreakers** — most importantly a hardcoded admin account seeded into *every* deployment that can't be disabled via the API, and an escapable code sandbox that yields RCE on the app host.
2. **A structural ops/data-integrity gap** — the Alembic migration chain cannot rebuild the schema; `create_all` is the real source of truth, so migrations drift from models by design and can never `ALTER` an existing table.
3. **A ring of "frontier / 10x" features that are theater** — the "AI Foundry that manufactures models," "Enterprise Physics," evolution "fitness," and the red-team "scans" are hardcoded constants, toy simulators, or dead code dressed as autonomous AI. Combined with marketing overclaims ("production-ready," fabricated telemetry in `skill.md`, self-graded `run_*.py` "verification" scripts), this is the biggest *credibility* risk for fundraising or enterprise sales.

**Composite score: 6.5 / 10** — a strong, genuine core (~7.5) pulled down by fixable-but-serious security, migration, and credibility gaps. The gap between the real engineering and the marketing narrative is the single biggest thing to close.

## 3. Dealbreakers (fix before ANY non-local deployment)

### D1 — [CRITICAL] Default admin `demo@kaeos.ai` / `demo123` seeded into every deployment, undeletable
`backend/app/main.py:114` → `backend/app/services/auth.py:111-145,290`
`AuthService.seed_demo_user()` is called **unconditionally** in lifespan startup — *not* gated on `DEV_MODE`. It creates a `role=ADMIN, tenant_id=tenant_acme` user with a public, well-known password. `/api/v1/auth/login` is public, and `deactivate_user` explicitly refuses `is_demo` accounts (`auth.py:290`), so the account **cannot be disabled via the API**.
**Exploit:** `POST /auth/login {"email":"demo@kaeos.ai","password":"demo123"}` on any instance → full admin token for `tenant_acme` (which also holds all the seeded demo data). *Independently confirmed during this audit.*
**Fix:** Gate the seed on `DEV_MODE`; or force a random password on create and log it once; and allow disabling demo accounts.

### D2 — [CRITICAL] Escapable AST sandbox → RCE on the app host
`backend/app/services/sandbox.py:19-73`, callers `backend/app/agents/mcp_tools_dynamic/registry.py:49-61`
`validate_ast` blocks disallowed *imports* and *direct name calls* (`eval`/`open`/…) but not attribute traversal — the classic `().__class__.__bases__[0].__subclasses__()[…]` reaches `os`/`subprocess` with no import and no blocked name. Validated code runs via `create_subprocess_exec(sys.executable, …)` as the same OS user with no container/seccomp/namespace and no memory/FS limits (the configured `AGENT_SANDBOX_MEMORY_MB`/`TIMEOUT_SEC` are **not enforced**). These are *agent tools* (`python_sandbox`, `calculator`), so any untrusted content reaching an LLM tool-call → RCE.
**Fix:** Replace with real isolation (container/gVisor/seccomp or RestrictedPython) and enforce the configured limits.

### D3 — [CRITICAL] Governance gates fail *OPEN* in degraded / no-provider mode
`backend/app/services/llm_router.py:462-546`
With no LLM reachable, `_simulated_completion()` returns canned **PROCEED** payloads: fairness → `overall_score: 0.95`, debate arbitrator → `decision: "PROCEED", final_confidence: 0.9`, compliance → `[]`. The fairness/debate/HITL gates then auto-pass. Payloads carry `"simulated": True` (honest) but **nothing downstream blocks on that flag** — a headless deploy silently rubber-stamps exactly the Tier-1 actions the gates exist to catch. For a product whose whole thesis is *governed autonomy*, this is a thesis-level defect.
**Fix:** In non-dev, treat `simulated` outputs as fail-*closed* at every gate.

### D4 — [CRITICAL/OPS] Alembic cannot build the schema; migrations drift from models by design
`backend/app/core/database.py:230-251`, `alembic/versions/8839c5a0b56c_initial_migration.py`
The "initial" migration creates **one** table; the ~175-table schema is materialized at runtime by `Base.metadata.create_all`. `alembic upgrade head` on an empty DB yields an almost-empty DB (documented bootstrap order is `init_db()` *then* migrate). `create_all` never issues `ALTER`, so a changed column type/constraint on an existing table is **never applied** to a live DB. No reproducible schema history, no real drift detection.
**Fix:** Autogenerate a real baseline migration from the models and make Alembic the source of truth; add CI drift check.

### D5 — [HIGH] `/pipeline/run` resolves tenant then discards it — untenanted ingestion
`backend/app/api/routes/pipeline.py:22-33` vs `backend/app/services/pipeline_service.py:23-31`
The one path that pulls **external data in** (extract→transform→load, incl. signal ingestion) never passes `tenant_id` to `PipelineService.run_pipeline` (which has no such parameter). Isolation depends entirely on the `current_tenant_id` contextvar being set — records not carrying `tenant_id` are untenanted. Postgres RLS backstops in prod; SQLite (default) does not.
**Fix:** Thread `tenant_id` through and tag every ingested record.

### D6 — [HIGH] RLS is inert on the shipped default; isolation is single-layer, not defense-in-depth
`backend/app/core/rls.py:29`, `backend/app/core/config.py:22`
RLS only applies on Postgres **and** only when the app connects as the non-owner `kaeos_app` role. The default `DATABASE_URL` is SQLite (dev/CI), where RLS no-ops. If prod points at the owner role (a common misconfig — owners bypass their own policies), every policy installs but is inert, and the hand-written `.where(tenant_id==…)` filters become the *only* isolation. `scripts/verify_rls.py` exists but is not a startup gate.
**Fix:** Refuse to boot in prod on SQLite or as the owner role; wire `verify_rls` into startup as a hard gate.

## 4. High-value findings by area

### Security & tenancy (beyond dealbreakers)
- **[HIGH] Custom hand-rolled auth token mislabeled "JWT"** — `services/auth.py:72-104` is `base64(json).hmac_sha256`, not a JWT. Cryptographically sound (constant-time compare, expiry) but no revocation, no `alg`/`iss`/`aud`, so a compromised token is valid until 24h expiry. Prefer `pyjwt`.
- **[HIGH] Admin API-key bootstrap outside tenant auth** — `main.py:286-311` mints admin keys for *any* tenant, guarded only by one static `X-Admin-Secret`. Mitigated by fail-fast config checks + `include_in_schema=False`, but needs network ACLs the code can't enforce.
- **[MED] BYOK at-rest key = unsalted `sha256(SECRET_KEY)` with dev fallback `"kaeos-dev-secret"`** — `services/live_connectors.py:55-59`. No entropy check on `SECRET_KEY`; rotating it silently breaks decryption of every stored customer key.
- **[MED] Weak staging secrets on disk** — `.env.staging` ships `JWT_SECRET_KEY=stg_secret_key_1234567890abcdef` and a real `POSTGRES_PASSWORD`. (Gitignored & untracked — confirmed — but rotate the DB password.)
- **[MED] `DEV_MODE` bypass guard keys only on `ENVIRONMENT` string** — `tenant.py:105-113`; an `ENVIRONMENT=dev`/`local` + `DEV_MODE=true` deploy boots with zero auth and `X-Tenant-ID` impersonation.
- **[MED] `predictive.analyze_and_predict` reads a Signal with no tenant filter** — `routes/predictive.py:19`; cross-tenant leak on non-RLS (SQLite) deploys.
- **[LOW] In-memory rate limiter** (per-process, resets on restart), **hardcoded RLS role password fallback** `kaeos_app_dev` (`database.py:207`), **JWT in localStorage + WS URL query string** (log-leak surface).

### Data layer
- **[HIGH] `tenant_id` is never a FK; no `tenants` table** — 209 FKs, 0 on `tenant_id`. Typo'd/stale tenant IDs create silently-orphaned rows; under RLS they become invisible (integrity bug → silent data loss). No cascade on tenant offboarding.
- **[HIGH] Polystore has no `tenant_id` column** — `core/polystore.py:88-97`; isolation is a caller-chosen collection-name prefix (the single current caller does it right, but nothing enforces it). Also: it's TF-IDF keyword matching, not embeddings, despite the "vector store" label.
- **[HIGH] Seeding runs on every startup and injects fabricated business metrics** ("seeded fiction") — e.g. `execution_count=1847`, `events_ingested=48210`, payroll `total_gross=412_500.00`. Mitigated: it never overwrites real data (short-circuits if data exists; all hardcoded to `tenant_acme`), so the risk is `tenant_acme` being demoed *as if* live — not destruction. But `top_up_new_entities` re-asserts fictional finance rows on every boot.
- **[MED] `provenance_ledger.tenant_id` is nullable** — the only nullable tenancy key; under RLS a NULL-tenant row is both invisible and un-writable, a hole in the ledger's own guarantee.

## 5. What's genuinely strong (credit due)

- **The governance spine is real:** 6-gate pipeline (compliance→fairness→confidence/HITL→debate→exec→audit), HITL that *actually blocks* (`PENDING_HITL` before execution, Redis-persisted, cross-tenant approval denied), real adversarial debate, real fairness audit trail.
- **LLM gateway is the strongest artifact:** multi-provider BYOK, tenacity retry, circuit breaker, tier fallback, pre-dispatch budget gate, honest per-call cost metering that reports `metering_active:false` rather than inventing a bill.
- **Real data, honestly benchmarked:** actual Kaggle enterprise datasets, code-computed metrics that *openly report losses*, `hours_saved`/`cost_reduction` returned `null` with a note rather than fabricated.
- **Frontend is a functioning product,** not a prototype — real JWT auth, polling, WebSocket with backoff, lifecycle state machines, file export.
- **Prior critical bugs genuinely fixed:** the cross-tenant IDOR (`tenant_id="default"` query param) is gone; the flagship `SyntaxError` is gone; the MCP plaintext key leak is closed.
- **Engineering docs are unusually candid** — `IMPLEMENTATION_STATUS.md` documents its own holes and retracts a premature "production-ready." The dishonesty gradient runs *upward* toward marketing, not down in the engineering record.

## 6. Prioritized remediation roadmap

**P0 — Security dealbreakers (days):** D1 (demo admin), D2 (sandbox), D3 (fail-closed gates), D5 (pipeline tenant), then the custom-JWT/admin-bootstrap/BYOK-key items in §4.

**P1 — Integrity & isolation (1–2 weeks):** D4 (real migration baseline + CI drift gate), D6 (refuse SQLite/owner-role in prod + wire `verify_rls` gate), `tenant_id` FK + `tenants` table, polystore tenant column.

**P2 — Credibility cleanup (1–3 days, high ROI):** delete `run_*.py` fabricated-verdict scripts + their JSON; fix `skill.md` telemetry (label as illustrative or derive from real rows); soften README "production-ready" → "beta / re-verification in progress"; report benchmark losses honestly; remove/`docs/`-archive `self_assessment.md` + `implementation_plan.md`; fix the 426→419 badge.

**P3 — Relabel the theater (ongoing):** either implement or clearly mark as "roadmap/simulation" the Foundry "manufactures models," Enterprise Physics, evolution fitness, and polymorphic red-team surfaces. Fabricated confidence values (`regulatory 1.0`, elicitation quality) should be computed or removed.

**P4 — Quality (ongoing):** type the frontend API surface (kill `any`), add mutation error UI, responsive breakpoints; consolidate JSON parsing onto `json_utils`; fix N+1 + silent-zero exception swallowing.
